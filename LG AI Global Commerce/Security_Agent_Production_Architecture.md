# 실제 프로덕션 환경에서의 'Security Guardian' 동작 원리

이 문서는 해커톤 데모용 시뮬레이션이 아닌, **실제 상용화(Production) 환경에서 AI 기반의 자율형 보안 에이전트(Autonomous Security Agent)가 어떻게 동작하는지**를 설명하는 아키텍처 개요입니다. 비싼 외부 보안 솔루션 없이도 MSA 아키텍처 내에서 자체적으로 구현 가능한 초경량 AI 보안 체계를 목표로 합니다.

---

## 1. 감지 (Perception - 실시간 로그 수집)
- **개요:** 외부의 상용 보안 솔루션(Datadog, Cloudflare 등)에 의존하지 않고, 내부 시스템에서 발생하는 트래픽을 스스로 읽어 들입니다.
- **구현 방식:** 
  - 백엔드 API 서버(Spring Boot / Node.js) 앞단에 **보안 필터(Security Middleware)**를 설치합니다.
  - 모든 HTTP 요청의 IP 주소, 요청 빈도(Rate), 페이로드(Payload)를 가로채어 실시간으로 인메모리 DB(예: Redis)나 이벤트 스트림(Kafka)으로 전송합니다.

## 2. 지능형 분석 (Analysis - LLM 문맥 이해)
- **개요:** 기존의 정적 룰 엔진(Rule-based WAF)이 막지 못하는 신종 공격이나 교묘한 변형 페이로드를 AI가 문맥적으로 분석합니다.
- **구현 방식:** 
  - 단순 매크로는 룰 엔진이 차단하지만, 모호한 특수문자의 조합이나 비정상적인 데이터 접근 시도는 **Gemini AI 모델**로 전송됩니다.
  - AI는 "이 페이로드가 SQL Injection 공격인지, 아니면 정상적인 검색어 입력인지"를 문맥(Context)을 바탕으로 정확히 판단합니다.

## 3. 자동 대응 (Action - Tool Calling 및 방어 조치)
- **개요:** AI가 위험을 감지하면 사람을 기다리지 않고 스스로 즉각적인 방어 조치를 취합니다.
- **구현 방식:** 
  - AI 에이전트에게 **Function Calling(도구 호출)** 기술을 연동합니다.
  - 위협 판단 시, AI가 직접 `block_ip()`, `update_waf_rule()`, `enable_rate_limit()` 등의 백엔드 방어 함수를 스스로 실행하여 서버 레벨에서 트래픽을 차단합니다.

## 4. 휴먼 인 더 루프 (Human-in-the-Loop) & 승인
- **개요:** AI의 오탐(False Positive)으로 인한 치명적인 서비스 장애를 막기 위한 안전장치입니다.
- **구현 방식:** 
  - 대규모 대역 차단 등 서비스에 큰 영향을 미치는 'Critical Action'의 경우, AI가 조치 전에 Slack 알림이나 어드민 패널을 통해 관리자의 승인(Approve)을 요청합니다.

## 5. 사후 보고 (Reporting)
- **개요:** 방어 조치가 끝난 후, 경영진 및 관리자가 알아보기 쉽도록 사건의 개요와 조치 내역을 문서화합니다.
- **구현 방식:** 
  - 에이전트가 방어 과정의 모든 로그를 종합하여 Gemini API에 전달하고, 자동으로 **"마크다운 형태의 전문적인 보안 사고 분석 보고서"**를 작성하여 발송합니다.
